# ADR 0020: Encore-generated TypeScript client as the canonical API client

## Status
Accepted

## Context
`apps/web` and `apps/mobile` need to call `apps/api`'s endpoints. Three shapes for the client layer:

- **Raw `fetch()` + manual paths + manual auth headers**, with `@rhitta/contracts` Zod schemas applied to request and response bodies.
- **Hand-rolled typed wrapper** (e.g., `apps/web/src/lib/api/notes.ts` with `createNote(...)`, `listNotes(...)` functions) that calls fetch internally and uses contracts schemas.
- **Encore-generated client SDK** — Encore.ts ships an `encore gen client` command that emits a fully-typed TypeScript client reading the endpoint declarations directly.

Encore's generated client is interesting because:
- It reads our existing concrete `*HttpRequest`/`*HttpResponse` interfaces (which our compile-time drift guards [ADR-0017 addendum] already keep in sync with the Zod schemas).
- Auth headers, paths, and error shapes are wired automatically.
- The client output is regeneratable from the source of truth at any time — a single command keeps consumers honest.
- Future endpoint additions in `apps/api` flow to consumers via the same regeneration step.

The downside: the client is generated, so consumers can't customize it inline. That's a feature, not a bug — customization belongs in app-side wrappers around the generated calls.

## Decision
`apps/web` (and Phase 2c `apps/mobile`) consume `apps/api` exclusively via the **Encore-generated TypeScript client SDK**. The generated client lives at:

```
apps/web/src/lib/api-client/
├── api-client.ts              # generated by `encore gen client`
└── README.md                  # explains regeneration command
```

Regeneration command: `pnpm --filter @rhitta/web gen:api-client` (script that wraps `encore gen client --output apps/web/src/lib/api-client/api-client.ts`).

Calling convention from app code:
```ts
import { client } from '~/lib/api-client'

const notes = await client.notes.list({ limit: 20 })
```

Wrapping the client in TanStack Query hooks lives at `apps/web/src/lib/queries/<domain>.ts` (e.g., `useNotes()`, `useCreateNote()`). These are the only API call sites in components — components never import the generated client directly.

**Hand-rolled fetch wrappers and raw `fetch()` calls to the API are banned.** Enforced by `@rhitta/biome-config/web-app`'s `noRestrictedImports` (e.g., bare `fetch` calls to `/api/*` paths flagged at lint time).

## Consequences
- **Zero hand-maintained client code.** Endpoint additions propagate via a regeneration step.
- **Types stay in sync** with the API by construction. The compile-time drift guards (ADR-0017 addendum) keep the API-side interfaces honest; the generated client reads them.
- **Auth + paths come for free.** No manual header threading.
- **Cost: an extra build step.** `gen:api-client` runs on local dev, in CI before typecheck, and in the production build. CI guarantees the checked-in client matches the API source.
- **Cost: the generated file is checked into the repo.** Diffs in `api-client.ts` are PR-visible — consumers see exactly what changed in the API surface. This is good. But the file is sometimes large; we accept the diff noise.
- **Customization happens in wrappers** (`lib/queries/`), not in the generated client. Cache policies, retry logic, optimistic updates — all live in TanStack Query wrappers.
- **Mobile mirror.** Phase 2c's `apps/mobile/src/lib/api-client/` follows the identical pattern. One mental model across platforms.
- If we ever need a non-Encore consumer (a future Rhitta tool, a downstream Rhitta consumer's own client), the generator can target other languages too — but that's not in scope for v0.
