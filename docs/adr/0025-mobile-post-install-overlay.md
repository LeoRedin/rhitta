# ADR 0025: Mobile scaffold is `npx ignite-cli new` plus a post-install overlay script

## Status
Accepted

## Context
Rhitta needs a mobile app shell with the same conventions as the rest of the monorepo — single auth gate (ADR-0023), Zustand for UI state (ADR-0024), TanStack Query for server state, TanStack Form everywhere, the centralized realtime hook (ADR-0021), the three-shapes-per-entity discipline (ADR-0013), and the `@rhitta/biome-config/mobile-app` variant.

Ignite is the locked starting point (per Phase 0 handoff). It ships an opinionated React Native + Expo template with screens, navigation, MobX state, an HTTP layer, and an i18n config. Most of those opinions don't match Rhitta's. We need to override.

Three integration shapes considered:

1. **Fork Ignite.** Maintain a long-running fork that ships Rhitta's conventions baked in. Pro: deterministic generator. Con: every Ignite upstream change requires rebase work; the fork's maintenance cost grows over time and is paid by Rhitta's maintainer.

2. **Generator fork or alternative CLI.** Write a Rhitta-specific RN generator. Pro: full control. Con: replaces an ~80%-fit tool with one we maintain; loses Ignite's community.

3. **Use `npx ignite-cli new` upstream, then run a post-install script that modifies the generated files.** Pro: Ignite stays in user's `package.json` as an external dep; upstream updates propagate via re-running the script against newer Ignite output. Con: the script lives in `apps/mobile/scripts/post-install.sh` and must stay current with Ignite's generated file shapes.

The user explicitly chose path 3 during the Phase 2c grilling session. This ADR records the choice and shape.

## Decision

Mobile scaffold lives at `apps/mobile/`. The directory is generated by Ignite, then patched by a Rhitta script.

### The flow

1. **First-time scaffold** (run by the Rhitta maintainer, not by Rhitta consumers):
   ```bash
   cd apps/
   npx ignite-cli@<pinned-version> new mobile --bundle com.rhitta.app --no-git
   bash apps/mobile/scripts/rhitta-overlay.sh
   ```

2. The `rhitta-overlay.sh` script applies the **Rhitta conventions overlay**. It is idempotent — running it twice produces the same result.

3. Subsequent Ignite upgrades:
   ```bash
   # 1. Diff what Ignite changed across versions; manually patch breaking changes
   # 2. Re-run rhitta-overlay.sh
   bash apps/mobile/scripts/rhitta-overlay.sh
   ```

### What the overlay does

Concretely (executed in this order; idempotent at every step):

1. **Switch routing to Expo Router** (per ADR-0023). Removes Ignite's `app/navigators/`, adds `app/_layout.tsx`, `app/index.tsx`, `app/_authenticated/_layout.tsx`. Updates `package.json` to add `expo-router` and remove `@react-navigation/native-stack` direct usage where it was opinionated.

2. **Strip MobX-State-Tree** (per ADR-0024). Removes `app/models/` and its barrel; removes MobX/MST runtime deps; replaces store-wrapping HOCs in screen scaffolds.

3. **Replace Ignite's HTTP layer with the shared Encore client.** Removes `app/services/api/`; the equivalent lives at `apps/mobile/src/lib/api-client/` (mirroring `apps/web/src/lib/api-client/`) and is regenerated from the same Encore source.

4. **Wire `@rhitta/design-tokens` into Ignite's themed factory.** Rewrites `app/theme/colors.ts`, `spacing.ts`, `typography.ts` to import semantic tokens from `@rhitta/design-tokens/semantic`. Removes Ignite's hardcoded palette.

5. **Inject Rhitta config files.** Drops in `apps/mobile/tsconfig.json` (extends `@rhitta/tsconfig/mobile.json` — possibly a new `@rhitta/tsconfig/mobile-app.json` variant if Expo's TypeScript profile needs relaxations, mirroring Phase 2a's api-app variant), `apps/mobile/biome.json` (extends `base + react + mobile-app`), and folder skeletons for `src/lib/{api-client,auth,queries,realtime,theme,toasts}`.

6. **Update `apps/mobile/package.json`** with: pinned versions (Expo SDK, RN, Expo Router, TanStack Query/Form, Better Auth, Zustand, design-system-mobile workspace dep, contracts workspace dep, Anthropic SDK if used directly). Sets `"name": "@rhitta/mobile"`, `"private": true`.

7. **Patch metro.config.js** for monorepo resolution (Expo + pnpm needs workspace resolution config — Metro defaults don't traverse pnpm symlinks correctly).

8. **Delete Ignite's example screens** (`app/screens/WelcomeScreen.tsx`, etc.). Reference Notes screens land in a separate task.

9. **Idempotency guard.** The script writes `apps/mobile/.rhitta-overlay-applied` containing the overlay version + timestamp. Re-running the script reads this marker; if the version matches, individual file patches are still safe (they check before applying) but the script reports "already applied" and exits 0 cleanly.

### Why a script and not a Rhitta CLI fork

- Ignite's update cadence outpaces what one Rhitta maintainer can keep up with via a fork.
- A script is read-and-understand-once; a fork is read-and-understand-every-upstream-change.
- The script's behavior is auditable and reversible (run git revert after).
- The script lives in the repo; Rhitta consumers see exactly what was applied.

## Consequences

- **Ignite upgrades go through:** new Ignite, re-run overlay, verify the overlay's individual patches still find their target files (the script must fail loudly if a target file moved upstream).
- **Cost: the overlay script must stay current with Ignite's file layout.** When Ignite renames `app/navigators/` to `app/navigation/`, the script must be updated. This is the explicit maintenance burden.
- **Cost: the script is bash + sed/awk + Node helpers.** No TypeScript runtime for the overlay itself; the script does text patching and JSON manipulation directly. Acceptable — overlay logic is small.
- **Cost: every overlay step is documented in the script** with a brief comment explaining *why*. The script is the de facto documentation; readers should be able to understand the deviation from upstream Ignite without ADR-0023/0024 in front of them.
- **The overlay applies once.** Subsequent edits in `apps/mobile/` are normal source code under the conventions Rhitta enforces. Future Phase 3 generators (`gen:module`, `gen:resource`) target the post-overlay shape.
- **For Rhitta consumers running `create-rhitta` (Phase 3):** the CLI runs `npx ignite-cli new` + `rhitta-overlay.sh` as part of the scaffold step. The user never sees Ignite's pristine output; they see Rhitta's conventions from the first commit. This is documented in Phase 3's handoff.
